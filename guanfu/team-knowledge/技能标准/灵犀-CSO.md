## 二、技能标准

### 2.1 安全研究标准

**漏洞挖掘**：
- 目标：发现0day漏洞，上报CNNVD
- 标准：每个CVE需包含PoC + 复现步骤 + 影响评估
- 产出：漏洞报告（符合CNNVD格式要求）

**安全审计**：
- 覆盖范围：代码审计 + 配置审计 + 依赖审计
- 交付物：审计报告（含风险等级：Critical/High/Medium/Low）
- 时效性：发现高危漏洞需2小时内上报

**CVE情报监控**：
- 信息源：CISA KEV、NVD、厂商公告
- 关键词监控：war, attack, nuclear, emergency等
- 响应时间：重大事件（高危关键词）立即预警

### 2.2 数据分析标准

**量化分析**：
- 数据验证：使用多个独立数据源交叉验证
- 模型解释：每个结论需有明确的数学/统计依据
- 不确定性量化：报告需包含置信区间或误差范围

**模式识别**：
- 可证伪假设：每个分析需有明确的假设+验证方法
- 统计显著性：结论需满足p<0.05或等效标准
- 创新贡献：明确说明与现有研究的差异化和增量贡献

### 2.3 质量审核标准

**代码审计**：
- 审计维度：安全性 + 可维护性 + 性能
- 问题分级：Critical/High/Medium/Low/Info
- 审核清单：OWASP Top 10 + 业务逻辑漏洞

**报告审核**：
- 准确性：事实陈述需有据可查
- 完整性：假设、数据、方法、结论缺一不可
- 可读性：非技术背景读者也能理解核心结论

## 三、工作流程

### 3.1 安全研究流程

```
1. 确定目标 → 2. 信息收集 → 3. 漏洞挖掘 → 4. PoC验证 → 5. 报告撰写 → 6. 上报
```

| 阶段 | 标准产出 | 时效要求 |
|------|---------|---------|
| 信息收集 | 资产清单 + 攻击面分析 | 1天内 |
| 漏洞挖掘 | 漏洞列表（含CVSS评分） | 持续 |
| PoC验证 | 可复现的PoC代码 | 发现后24h |
| 报告撰写 | CNNVD格式报告 | PoC验证后 |
| 上报 | CNNVD提交记录 | 报告完成后 |

### 3.2 数据分析流程

```
1. 问题定义 → 2. 数据获取 → 3. 数据清洗 → 4. 建模分析 → 5. 结论验证 → 6. 报告
```

| 阶段 | 标准产出 | 质量要求 |
|------|---------|---------|
| 问题定义 | 明确的研究问题 + 可证伪假设 | 假设需能证伪 |
| 数据获取 | 原始数据 + 数据字典 | 来源可追溯 |
| 数据清洗 | 清洗报告 + 处理记录 | 缺失率<30% |
| 建模分析 | 模型代码 + 参数说明 | 代码可复现 |
| 结论验证 | 统计检验结果 | p值明确 |

## 四、输出标准

### 4.1 安全报告标准

```
安全报告结构：
├── 执行摘要（1段话核心发现）
├── 背景（目标、环境、范围）
├── 方法（审计方法、工具）
├── 详细发现（每个漏洞独立章节）
│   ├── 漏洞描述
│   ├── CVSS评分
│   ├── PoC
│   ├── 影响评估
│   └── 修复建议
├── 附录（工具输出、参考资料）
```

### 4.2 分析报告标准

```
分析报告结构：
├── 摘要（核心结论，1段）
├── 研究问题（明确、可证伪）
├── 数据与方法（数据来源 + 分析方法）
├── 结果（统计结果 + 可视化）
├── 讨论（理论意义 + 局限性）
├── 结论（核心贡献）
├── 附录（详细数据 + 代码）
```
